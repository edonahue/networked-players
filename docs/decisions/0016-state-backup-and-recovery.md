# ADR 0016: State backup and recovery for the coordination host

- **Status:** Accepted
- **Date:** 2026-07-02

## Context

Two real gaps have sat unchecked since Milestones 1 and 2 of
`docs/BUILD_PLAN.md`: "Write and locally test coordination-host recovery notes"
(Postgres/Redis) and "Back up manager state (Swarm CA/raft) and locally test its
recovery procedure." Nothing in the repo automated either — no backup/restore
scripts, no runbook.

Confirmed via research before deciding, not assumed:
- Postgres and Redis run as **plain `docker compose` containers**, not Swarm
  services (`infra/swarm/docker-compose.coordination.yml`), with volumes bound
  onto `/mnt/data/docker-volumes/{postgres-data,redis-data}`
  ([ADR 0013](0013-nvme-storage-layout.md)). Credentials live in the git-ignored,
  `chmod 600` `infra/swarm/.env`, generated by `deploy-coordination.sh`.
- `init-swarm-manager.sh` only runs `docker swarm init` and captures join tokens
  — it never touches `/var/lib/docker/swarm`, the directory holding the Swarm's
  CA and raft state. [ADR 0007](0007-zimaboard-swarm-manager.md)'s revisit
  trigger doesn't mention this either — a real gap, not a deferred decision.
- Backing up the Swarm raft store requires briefly stopping the Docker daemon —
  Docker's own documented constraint, since the raft store can't be safely
  copied while `dockerd` is writing to it. Stopping `dockerd` also stops the
  coordination-stack containers, and
  [ADR 0014](0014-coordination-host-hardening.md)'s own real incident already
  proved `restart: unless-stopped` does **not** reliably bring them back on its
  own after a daemon restart.
- The agent session that wrote this ADR has neither `docker` group membership
  nor `sudo` (confirmed live: `docker compose ps` failed with "permission
  denied... connect to the docker API" on the socket). Every script here has to
  be run by the operator, the same boundary that has held for every other
  real-infrastructure action in this project (`harden.yml`, `onboard.yml`,
  `init-swarm-manager.sh`).

## Decision

1. **Coordination-stack (Postgres/Redis) backup uses logical dumps, not raw
   volume tars.** `pg_dump` (transactionally consistent by design) and Redis
   `BGSAVE` (Redis's own async point-in-time snapshot), both taken through
   `docker compose exec`/`cp` rather than touching the bind-mounted volume
   directories directly. This needs no downtime and no root — it goes through
   the Docker socket via `docker` group membership, not filesystem access —
   and avoids the real risk of a torn/inconsistent backup from copying a live
   Postgres data directory with a plain `tar`.
2. **Swarm-manager state backup does need `sudo` and brief downtime.** Stop
   `dockerd`, `tar` `/var/lib/docker/swarm`, restart `dockerd` — there is no
   logical-export equivalent for the raft store the way `pg_dump` is for
   Postgres. Because this stops the coordination-stack containers too (see
   Context), the backup script automatically re-runs
   `deploy-coordination.sh`/`deploy-portainer.sh` afterward rather than relying
   on the operator to remember the ADR-0014 gotcha.
3. **Backups live under `local/backups/`** (already fully git-ignored) and are
   never committed. The Swarm tarball contains real cluster CA key material, so
   it is `chmod 600` immediately after creation.
4. **`restore-swarm-manager-state.sh` is gated behind an explicit
   `--yes-i-am-sure` flag** and moves (not deletes) the current
   `/var/lib/docker/swarm` to `swarm.bak` before extracting the backup — this is
   the single most destructive script in this pass, since it replaces the only
   Swarm manager's identity, and gets caution to match per this project's
   standing rule on hard-to-reverse actions affecting shared systems.
5. **Testing depth is asymmetric, matching blast radius.** The coordination
   stack is a low-stakes, currently-empty dev-loop database with no real
   consumers yet, so its recovery gets a real live round-trip test (write a
   marker, back up, restore, confirm the marker survives). The Swarm-manager
   restore only gets an automatic backup-integrity check (`tar -tzf`, confirming
   `raft/`, `certificates/`, and `docker-state.json` are present) — actually
   live-testing a full restore is left to the operator's explicit judgment,
   since this is the only Swarm manager that currently exists and a botched
   test has no fallback manager to recover from.

## Consequences

Backup/restore tooling now exists for both the coordination stack and the
Swarm manager, but no real backup has actually run yet as of this ADR being
written — `docs/BUILD_PLAN.md`'s Milestone 1 and Milestone 2 checkboxes stay
unchecked until the operator runs the scripts for real and confirms the
result, per this project's rule against claiming something exists before
there's evidence for it. Backups are local-only in this pass — nothing here
copies a backup off-host, so a total loss of `/mnt/data` (not just Docker
state) still loses everything; that's a real, stated limitation, not an
oversight, and is a reasonable candidate for its own future ADR if off-host
replication becomes a real priority.

## Validation

`bash -n` and `shellcheck` (where available) on all four new scripts;
`make check` (no Python code touched, expected no-op pass). Real execution —
an actual backup, a live restore round-trip for the coordination stack, and at
minimum a backup-integrity check for the Swarm manager — is the operator's to
run and confirm, since the authoring session has neither `docker` group
membership nor `sudo`.

## Revisit trigger

Revisit once the first real backup and restore actually run — confirm the
runbook in `docs/OPERATOR_SETUP.md` matches what actually happened, not just
what was planned. Revisit if the coordination stack stops being a low-stakes
empty dev-loop database (once `packages/workers`/`apps/api` start writing real
state to it) — the "live round-trip test is safe" reasoning in Decision 5
depends on that being true. Revisit if a second Swarm manager is ever added —
multi-manager Swarm has its own quorum/raft-recovery semantics this ADR
doesn't address, and the current single-manager assumption throughout this
document would need re-examining.
