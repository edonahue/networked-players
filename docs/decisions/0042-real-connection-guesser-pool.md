# ADR 0042: A real, Python-generated Connection Guesser pool replaces the synthetic-first one

- **Status:** Accepted
- **Date:** 2026-07-20
- **Extends:** [ADR 0037](0037-web-game-universe-and-round-engine.md) without reverting it
- **Relates to:** [ADR 0038](0038-hybrid-album-catalog-assembly.md), [ADR 0039](0039-performer-allowlist-layered-for-game-rounds.md)

## Context

ADR 0037 shipped the flagship Connection Guesser against a synthetic "Meridian
Tapes" universe plus a small (10-round) real-records pool, deliberately —
real album data (`challenge.v2.json`) didn't exist yet. Its own revisit
trigger named the moment this ADR now closes: "a live gate F real dump-derived
challenge... or a real `packages/game-rules`." `challenge.v2.json` is now real
(140 studio albums, `snapshot=20260601`) and the operator's product decisions
call for removing Meridian Tapes from all public production routes once the
real gate passes, keeping it only as an isolated test fixture.

## Decision

**The public pool is now 100% real.** `apps/web/public/data/game/universe.v1.json`
and `rounds.v1.json` are generated by a new Python command,
`networked-players-catalog build-connection-rounds`
(`packages/graph-core/.../connection_rounds.py`), and contain only
`pool: "real-records"` rounds. `apps/web/scripts/build-rounds.mjs` is demoted
to writing a synthetic-only **test fixture** at
`apps/web/tests/fixtures/game-{universe,rounds}.synthetic.v1.json` — never
`public/`. It no longer reads `challenge.v1.json` or derives a real pool at
all; that logic (`deriveRealRounds`) is deleted, superseded entirely by the
Python generator.

### Why a new generator, not `rounds_generator.py`

The Connection Guesser and `rounds_generator.py`/`rounds.py` (which now feeds
the separate Record Routes mode) answer **different questions**, discovered
differently:

| | Connection Guesser (`connection_rounds.py`) | Record Routes (`rounds_generator.py`) |
|---|---|---|
| One-hop question | A performer explicitly credited on **both displayed albums directly** | Artist A and artist B share a credit on some third release |
| Discovery | Album-keyed: index each album's own eligible performers, intersect album pairs | Artist-keyed: BFS/`neighbors_batch` over the credit graph |
| Answer | Every artist in the intersection (`answer_set`, possibly several) | The two path endpoint artists |
| Two-hop | A **hidden middle album** the player must also name, bridged by two independently-guessed performers | A two-hop graph path, both hops shown |

PR #43's original rounds pipeline assumed one album per artist (`albums_by_artist`
keyed by `artist_id`); the real 140-album catalog has multiple albums by the
same artist (Michael Jackson: *Thriller*, *Bad*, *Off the Wall*), which the
intersection semantic must key by album id instead. Reusing or bolting this
onto `rounds_generator.py` would have conflated two genuinely different
puzzles under one artifact shape. Both generators reuse the same policy
primitives (`eligibility.py`'s performer allowlist, the family-exclusion
callable) without reusing each other's discovery or assembly code.

### Runtime fetch, not a build-time JSON island

At real scale (500 rounds) `rounds.v1.json` is ~1.2 MB — far past a reasonable
inline-HTML budget. `game/flagship.ts` now fetches it at runtime
(`fetchRounds()`) instead of reading a `<script type="application/json">`
island baked into the page at build time; `initFlagship()` became `async`.
`universe.v1.json` is never fetched by any page at runtime at all (confirmed:
only `rounds.v1.json` is read) — it exists purely as a published, contract-
validated reference index, so its size is not a page-payload concern. A fetch
failure renders a graceful in-stage message rather than an uncaught error.

## Consequences

Real, measured (`snapshot=20260601`, 140-album catalog, family exclusion
applied): **409 one-hop candidates, 832 two-hop candidates found; 300 one-hop
and 200 two-hop selected** (500 rounds total) — 6x/10x the real-data-game
integration plan's launch floor (≥50 one-hop, ≥20 two-hop; distinct from ADR
0037's original ≥40 synthetic/≥6 real floor, which this pool now supersedes).
Universe scoped to the 101 albums actually referenced by a round
(endpoint, hidden middle, or a two-hop decoy choice). Every round is a real,
verifiable connection — e.g. one one-hop round names James Gadson (real
session drummer/producer) as a valid answer linking *Still Bill* (Bill
Withers) directly to *Futuresex/Lovesounds* (Justin Timberlake); a separate
two-hop round bridges *Still Bill* to *No Secrets* (Carly Simon) through the
hidden middle *Genius Loves Company* (Ray Charles), naming Gadson as the
first bridge and Jim Keltner (also a real session drummer) as the second.

- `apps/web/src/game/types.ts` widened to match real data: `RoleCategory`
  gained `vocals`/`strings`/`brass`/`woodwind`/`performer` (a fallback);
  `GameAlbum.year`/`.label` and `GameRelease.year` became nullable;
  `GameCredit.credit_scope` widened from a literal to `string` (real per-track
  credits use scopes beyond `release_credit`); `SyntheticProvenance` renamed
  to `GameProvenance` (it now holds real provenance too).
- `packages/graph-core/.../eligibility.py` gained `performer_role_category()`
  (a display-only categorization of the same token set, grouped identically)
  — used for chip labels; purely presentational, never branched on.
- `packages/graph-core/.../graph.py` gained `credit_rows_for_releases()`, a
  batched (one query, not one per release), eligibility-agnostic credit
  lookup over a set of release ids — `connection_rounds.py`'s only new
  dependency on `graph.py`, keeping the performer-allowlist boundary intact
  (`graph.py` still never imports `eligibility.py`).
- `apps/web/.prettierignore` added: `rounds.v1.json`'s (and the synthetic
  fixture's) nested-array shape trips Prettier's collapsing heuristic against
  `json.dumps`/`JSON.stringify` output — confirmed real formatting drift, not
  a mistake to fix by hand.

## Validation

`packages/graph-core/tests/test_connection_rounds.py` (11 tests: discovery
correctness, family exclusion, distractor-never-satisfies, universe scoping,
validator rejection cases) and `test_graph.py`'s new
`credit_rows_for_releases` tests (batching, placeholder exclusion,
non-playable-row exclusion). `apps/web/tests/game-data.spec.ts` re-verifies
the **real** pool independently in TypeScript against `challenge.v2.json`'s
own published credit rows (a from-scratch port of the performer-role
allowlist, not an import — a bug in the Python allowlist wouldn't be graded
by its own copy) and separately validates the **synthetic fixture** via the
demoted `build-rounds.mjs --check`. Full Playwright suite green (72/72)
against the real artifact with zero changes to existing engine/store/
flagship/twohop/daily/mobile specs — proving the swap required no game-engine
code changes, only a data-loading strategy change.

## Revisit trigger

If the real pool's floor (≥50 one-hop, ≥20 two-hop) ever becomes hard to
maintain as the album catalog is corrected or grows (e.g. a future
studio-album-policy tightening shrinks the backbone), diagnose the limiting
stage and propose the smallest principled catalog/policy expansion — never
weaken the performer allowlist or the family exclusion to hit a number.
Revisit the runtime-fetch decision if `rounds.v1.json` grows enough to need
sharding (e.g. by kind or by pool-version) rather than one fetched file.
