# ADR 0021: Canonical Pi benchmarks, and automated Dask/RQ contention isolation

- **Status:** Accepted
- **Date:** 2026-07-03

## Context

The only benchmark job body that existed (`benchmark_parse.py`) was a synthetic XML-parsing loop — useful for raw CPU/memory characterization, but not representative of what Pi workers are actually meant to do in production. `docs/DISCOGS_INGESTION.md`'s hardware execution profiles table names four canonical Pi job types (checksummed partition validation, role summaries, challenge batches, graph tests), none of which had a benchmark, and none of which have real production code yet either (`packages/workers` and `packages/graph-core` remain placeholder READMEs).

Separately, [ADR 0020](0020-dask-jupyter-interactive-cluster-compute.md) explicitly deferred "automatic contention-aware gating" between the manual, on-demand Dask worker path and real RQ production jobs sharing the same Pi, relying on an operator-checked manual rule instead. Real testing built enough understanding of both subsystems (RQ per-host queues, the Dask worker's real memory/version characteristics) to close that deferred item now.

## Decision

**Four new benchmark probes** (`infra/ansible/files/benchmark_validate.py`, `benchmark_role_summary.py`, `benchmark_graph_challenge.py`), stdlib+`duckdb`-only, each modeling one canonical job type's real *shape* against an in-memory, DuckDB-SQL- or stdlib-generated synthetic fixture — never a real Parquet partition (keeps them reproducible from source alone, per AGENTS.md's fixture rule, and avoids a new per-run file-copy payload to each Pi) and never an import of `packages/catalog` or `packages/graph-core` (would drag `lxml`/`pyarrow` onto a Pi, or half-build a package that doesn't exist yet). Sizing is grounded in real observed evidence already in the docs (the real chunk-size default, the real credits/release ratio, the real role-text cardinality from a full-dataset profiling pass) but explicitly labeled projected/matched-to-a-default where no real measurement at this scale exists — never asserted as itself measured.

`scripts/cluster_benchmark_distributed.py` gained a `JOB_KINDS` registry and a `--kind` flag selecting which probe runs; the enqueue/split/wait/aggregate pipeline was already kind-agnostic (it only ever treated the job function as an opaque string), so this was additive, not a rewrite. Output filenames now include the kind so different kinds don't clobber each other's `local/benchmarks/` output.

**OS-level priority isolation, `systemd-run --user` not system-level.** `scripts/run-ingest-supervised.sh`'s proven `Nice=`/`IOSchedulingClass=`/`MemoryMax=` pattern needs `sudo systemd-run`, which would force `--ask-become-pass` onto every RQ burst-worker invocation and break `cluster_benchmark_distributed.py`'s non-interactive driver outright (it has no path to supply a sudo password). `systemd-run --user` needs no root at invocation time — only a one-time `loginctl enable-linger <user>` (added to `harden-workers.yml`, which already does root-required OS configuration on these hosts) so the `--user` systemd instance survives a non-interactive SSH exec with no login session of its own. Confirmed live this needed no interactive sudo prompt on the real fleet (passwordless sudo already configured there for hardening tasks).

RQ (the project's stated production job path) runs at `Nice=5`/`IOSchedulingPriority=4`, close to normal priority — it should win CPU/IO contention. The Dask burst worker runs at `Nice=15`/`IOSchedulingPriority=7`, matching `run-ingest-supervised.sh`'s own choice for "yield to everything else" background work — Dask is explicitly the most optional workload on a Pi (ADR 0020: "an optional experiment"). `MemoryMax=512M` (RQ) and `600M` (Dask, above Dask's own internal 400MiB soft limit) are cgroup backstops, not claims about typical usage.

**Real bug found and fixed along the way:** `systemd-run` does not inherit the invoking process's environment into the transient unit it creates (unlike a plain `ansible.builtin.command`, which execs directly and inherits normally). The first live test failed every job with "Invalid attribute name: benchmark_validate.run_benchmark" — RQ couldn't find the probe module — because `PYTHONPATH` never reached the unit. Fixed with an explicit `--setenv=PYTHONPATH=...` on the `systemd-run` command line; the `environment:` key on the Ansible task still sets `XDG_RUNTIME_DIR` for `systemd-run` itself, a separate concern from what environment the unit it creates then runs with.

**Automated RQ-queue-depth gating: both a start-time gate and an ongoing watcher, layered with the priority scheme above.** They cover different failure windows — the gate stops a Dask worker from starting onto an already-busy Pi; the watcher covers an RQ job arriving *after* Dask is already running (e.g. during an open-ended Jupyter session) — and the OS-level niceness is what protects the window before the watcher's next poll. New `infra/ansible/files/dask_gate_check.py` reuses the exact `Queue`/registry API `cluster_benchmark_distributed.py`'s `assert_queue_empty()` already uses, discovering every queue name that has ever targeted this host via RQ's own persistent `rq:queues` Redis set (confirmed live this set survives a queue being fully drained, unlike the per-queue list key) rather than a fixed list of known prefixes — a future production queue name doesn't silently bypass the gate. `check` exits 0 (idle)/1 (busy)/2 (broker unreachable); `watch` polls every 20s and stops the named `--user` units on the first busy observation.

**Fail-open when the jobs broker is unreachable, deliberately.** The broker (`infra/swarm/docker-compose.jobs-broker.yml`) is not a standing service — it's only up around a benchmark run. If it's down, no RQ job can possibly be running anywhere; fail-closed would block ordinary Dask/Jupyter use in the common case (broker off because nobody's benchmarking) for a contention risk that provably cannot exist.

## Consequences

Five job kinds now share one benchmark driver and one deploy playbook — adding a sixth later is additive (one new probe file, one new `JOB_KINDS` entry), not a parallel pipeline. RQ burst workers and the Dask burst worker both now run inside transient `systemd-run --user` units instead of a bare foreground command / `nohup`, which also means their output is durably in `journalctl --user`, not just a scratch log file for Dask's case (a real improvement, not just a side effect of the isolation work). `run-dask-worker-burst.yml` gained two new required preconditions (`jobs_broker_url`, matching its RQ sibling, and a live gate check) — starting a Pi Dask worker now genuinely fails loudly if the fleet is busy, rather than relying on an operator remembering to check first.

This does not fully solve memory contention — the OS-level niceness only affects CPU/IO scheduling, not memory allocation, so a genuinely simultaneous memory-heavy RQ job and Dask task on the same Pi could still both fit within their respective `MemoryMax` ceilings independently while collectively pressuring the host; the up-to-20s watcher window is the accepted gap this pass leaves open, informed by real experience rather than guessed at.

## Validation

All confirmed live, 2026-07-03, against the real 3-worker fleet:
- All four new probes ran locally first, then deployed and executed via RQ against real workers — real per-worker fields (hostname, elapsed, throughput, peak RSS) in each run, not placeholders.
- `systemd-run --user` confirmed working with no interactive sudo prompt, after the one-time `loginctl enable-linger` ran cleanly via the existing `--ask-become-pass` hardening flow.
- Priority isolation observed at the kernel level, not just configured: `ps -o pid,ni` showed `NI=5` for a real running RQ burst-worker process, and its cgroup's `memory.max` file showed exactly `536870912` (512MB) — both matching the configured `systemd-run` properties exactly.
- Start-time gate: enqueuing a real job on a Pi's queue and then attempting to start a Dask worker there failed at the gate-check task, before any Dask installation happened.
- Ongoing watcher: starting a Dask worker on an idle Pi (confirmed registered with the scheduler), then enqueuing and running a real RQ job on that same Pi, resulted in the watchdog detecting the job and cleanly stopping both `dask-worker-burst` and `dask-worker-watchdog` within about 12 seconds — well inside the ~20s poll interval, confirmed via `journalctl --user -u dask-worker-watchdog`.
- `make check` (ruff, format, mypy, pytest) stayed green throughout; the new probe/gate-check files are in `ruff`'s scope but outside mypy's `packages = ["networked_players_catalog"]` scope and pytest's `testpaths`, the same pre-existing gap other `infra/ansible/files/*.py` already have, not a new regression.

## Revisit trigger

Revisit the memory-contention gap (both processes independently under their own `MemoryMax` but collectively pressuring the host) if real experience shows it matters in practice — no data exists yet either way. Revisit the ~20s poll interval if real usage shows it's too slow or unnecessarily aggressive. Revisit whether any of the four new benchmark kinds should eventually be replaced by real production code once `packages/workers` or `packages/graph-core` gain one, at which point the synthetic probe and the real implementation should be benchmarked side by side before retiring the synthetic one.
