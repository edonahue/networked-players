# ADR 0024: Read-only HTTP access to catalog datasets for remote workers

- **Status:** Accepted
- **Date:** 2026-07-04

## Context

The real parsed datasets live in exactly one place: the coordination host's NVMe, under git-ignored `local/processed/`. The standing Dask stack ([ADR 0020](0020-dask-jupyter-interactive-cluster-compute.md)) bind-mounts that tree into its own containers — but a bind mount is host-local, so the on-demand remote workers ([ADR 0023](0023-x86-worker-joins-rq-dask-fleet-work.md)) have no path to the data at all. This is not hypothetical: the first time the x86 worker registered with the scheduler during a real-data notebook run, it was assigned a partition-read task and failed with a reproduced `FileNotFoundError` on a file that exists only on the coordination host. The RQ path ([ADR 0019](0019-cluster-benchmark-rq-job-broker.md)) deliberately sidesteps this with self-contained synthetic job bodies, but genuine distributed compute over the real dataset needs real data access.

Milestone 5's one-hop corpus (`data/contracts/discogs-onehop-v1.md`) sharpens the need and the shape: a small, seed-derived dataset that bounded workers should read without each getting a private copy.

## Decision

**Serve `local/processed/` read-only over HTTP from the coordination host** (`infra/swarm/docker-compose.catalog-data.yml`: nginx:1-alpine, ~tens of MB RSS, `mem_limit: 128m`), following the jobs-broker deployment pattern exactly — bound to the real LAN interface (never `0.0.0.0`), env in git-ignored `local/catalog-data.env` generated by `deploy-catalog-data.sh`, and deliberately **not** a standing service: started before a remote-worker session, stopped after.

Why HTTP over the alternatives considered:

- **NFS — rejected outright.** Client mounts need host packages via apt, which is forbidden on the x86 worker (fragile dpkg state, ADR 0023) — disqualifying it for the one host that matters most.
- **MinIO/S3 — rejected for now, recorded as the revisit path.** Hundreds of MB RSS on a shared host, credential distribution that would be theater on a LAN where the Dask scheduler's port 8786 is already unauthenticated, and no measured need for S3 semantics (no writes, no multipart, no versioning beyond the existing `snapshot=` convention). AGENTS.md requires a measured need for a new stateful service; none exists.
- **rsync replication — kept as a documented zero-infrastructure fallback**, reasonable for the small one-hop dataset to a single host, but it multiplies storage and invents a staleness problem on every re-ingest.
- **HTTP wins on fit:** nginx's default static-file **range requests** are what make remote parquet reads efficient — DuckDB (`httpfs`) and PyArrow/fsspec fetch only the columns and row groups a query actually touches. And HTTP's classic weakness (no globbing) is already solved by this project's own design: every dataset ships a `manifest.json` naming each file with its sha256, so clients build exact URL lists with integrity metadata for free (`networked_players_catalog.discogs.dataset_locator.dataset_file_urls`, stdlib-only so it runs in the lean worker venvs).

**Hybrid access policy by hardware class** (operator decision): the x86 worker (1 GbE, 7.6GB RAM) may read partitions of the full dataset over HTTP; the Pi 3Bs (100 Mbps, 1GB RAM) only ever read the one-hop dataset or bounded partitions within the existing 128–256MB guidance — never the full dataset. Enforced by job design plus the existing per-class `MemoryMax` bounds (ADR 0023); this ADR states the rule so a future job that violates it is contradicting a recorded decision, not filling a gap.

**Consumption plumbing:** `run-dask-worker-burst.yml` accepts an optional `-e catalog_data_url=...`, delivered to worker tasks as `CATALOG_DATA_URL` via `--setenv` (systemd-run does not inherit the invoking environment — the same confirmed-live gotcha ADR 0021 recorded for `PYTHONPATH`), and the pinned Dask venv gains `aiohttp` (fsspec's HTTP backend) as the one new dependency.

**Evidence-driven split optimization:** `scripts/cluster_benchmark_distributed.py` gains `--worker-weights 'host=N,...'` for proportional chunk splitting (default unchanged: even). Weights are meant to be set from a real observed per-host throughput ratio out of `local/benchmarks/`, satisfying ADR 0023's proportional-splitting revisit trigger with data rather than guessed hardware specs.

## Consequences

A second LAN-reachable service joins the jobs broker, with the same mitigations (specific-interface bind, non-standing, no persistent state of its own — it serves existing read-only files). The trust boundary is honestly stated: an unauthenticated read of a private dataset, LAN-only — the same class as Dask's 8786. Anyone on the LAN can read the catalog data while the service is up; that data is CC0-sourced catalog content plus seed-*derived* selection (the one-hop corpus), which the operator accepts on a private home LAN with the service off by default. Remote Dask workers can now genuinely participate in real-data reads, closing the reproduced `FileNotFoundError` gap. The static-first public boundary is untouched — this is lab-internal plumbing, invisible to the public experience.

## Validation

`dataset_locator` is tested against a real local HTTP server over a synthetic dataset (URL list matches the manifest, fetched bytes hash-verify, unknown table and unreachable server fail loudly). `deploy-catalog-data.sh` passes shellcheck-equivalent bash syntax; compose file validated by deployment. The live gate: with the server up, a remote x86 Dask worker started with `catalog_data_url` participates in the real-data notebook (`02-explore-real-discogs-catalog.ipynb`) end-to-end — the exact scenario that previously failed.

## Revisit trigger

Revisit toward MinIO/S3 if any of: off-LAN access is ever needed, workers need write-back, or a measured need for S3 semantics appears. Revisit the unauthenticated-read posture if the LAN trust model changes (same trigger as Dask's 8786). Revisit the Pi bounded-only rule if a real Pi workload demonstrates a safe need for larger reads.
